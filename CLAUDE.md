# 프로젝트
[프로젝트 한 줄 설명]

## 권장 작업 흐름
- 새 기능, 큰 리팩터링, 범위가 큰 버그 수정은 먼저 Plan Mode로 조사와 계획을 수행한다.
- 구현이 끝나면 reviewer 기준으로 검토한다.
- 리뷰 피드백이 있으면 필요한 범위만 다시 수정한다.

## 공통 규칙
- 커밋 메시지는 Conventional Commits 형식을 사용한다.
- 주석과 커밋 메시지는 한국어로 작성한다.
- 코드, 식별자, 파일명은 영어를 사용한다.
- `any`, `# type: ignore` 같은 타입 회피는 마지막 수단으로만 사용한다.
- 비밀번호, API 키, 토큰 등 민감정보를 하드코딩하지 않는다.
- 기존 코드 스타일과 패턴을 우선적으로 따른다.

## 작업 원칙
- 수정 전에 관련 파일, 테스트, 기존 구현 패턴을 먼저 읽는다.
- 추측으로 구현하지 않는다.
- 서로 무관한 변경은 한 커밋에 섞지 않는다.
- 변경 후 가능한 범위에서 테스트, 타입체크, 린트를 수행한다.
- 에러를 덮어쓰지 말고 원인을 확인한 뒤 수정한다.

## Git 규칙
- 브랜치명:
  - `feature/<name>`
  - `fix/<name>`
  - `refactor/<name>`
